## 🕳️ 坑 1：Chrome VNC 黑屏（调试了 4 次的大坑）

### 现象
VNC 连上桌面后，能看到底部 xfce4-panel 图标，但 Chrome 浏览器窗口完全看不到（或显示为黑色方块）。

### 调试过程（4 次失败尝试）

| 尝试 | 参数 | 结果 | 原因 |
|------|------|------|------|
| 第1次 | `--disable-gpu` | ❌ 黑屏 | GPU 进程被完全杀死，Chrome 失去渲染管道 |
| 第2次 | `--use-gl=swiftshader` | ❌ 黑屏 | 系统无 SwiftShader 库，GPU 反复崩溃重启 |
| 第3次 | `--use-gl=angle --use-angle=swiftshader-webgl` | ❌ 黑屏 | 同上 |
| 第4次 | `--ozone-platform=x11 --use-gl=angle --use-angle=default` | ⚠️ 窗口存在但被放到虚拟桌面3 | `_NET_WM_DESKTOP = 3` |

### 根因（源码级）

```
Chrome GPU 进程默认 --ozone-platform=wayland
  → 渲染输出到宿主机真实桌面
  → VNC X11 虚拟桌面什么都没收到
  → 用户看到黑屏
```

系统 GL 实现检测：
```
Requested: gl=swiftshader → NOT FOUND
Available: gl=egl-angle, angle=default
```

### ✅ 最终解决方案

```bash
DISPLAY=:2 google-chrome-stable \
  --no-sandbox \
  --ozone-platform=x11 \
  --disable-gpu \
  --disable-gpu-compositing \
  --disable-software-rasterizer \
  --disable-features=VizDisplayCompositor \
  --remote-debugging-port=9222 \
  --remote-allow-origins=* \
  --user-data-dir=/tmp/chrome-xhs \
  --window-size=1920,1080 \
  --no-first-run \
  --no-default-browser-check \
  --new-window "https://www.baidu.com"
```

**成功三要素**：
1. `--ozone-platform=x11` — **强制 X11**，防止 GPU 偷跑 Wayland（最关键！）
2. `--disable-gpu --disable-gpu-compositing --disable-software-rasterizer` — **三连杀**，纯 CPU 渲染
3. `--disable-features=VizDisplayCompositor` — 关闭 Viz 合成器

> ⚠️ 仅 `--disable-gpu` 不够！必须同时指定 `--ozone-platform=x11`。
> ⚠️ Chrome 可能被放到虚拟桌面 3，用 xdotool 搬回桌面 0：
> ```bash
> DISPLAY=:2 xdotool set_desktop_for_window <窗口ID> 0
> ```

## 🕳️ 坑 2：Vue 自定义组件点不动（核心大坑）

### 现象
小红书的「发布」按钮 `<xhs-publish-btn>` 是一个 Vue 3 `defineCustomElement` 创建的自定义 Web Component。
- 用 CDP 鼠标事件点击 → 无效
- 用 `btn.click()` → 无效
- 用 `dispatchEvent(new MouseEvent('click'))` → 无效
- 元素内部 0 个子节点，没有任何可点击的 DOM

### 原因
Vue 3 的 `v-on:click` 处理器不是原生 DOM addEventListener，而是存在元素的 **`Symbol(_vei)`**（Vue Event Invokers）内部属性里。CDP 的鼠标事件在浏览器底层生成，不经过 Vue 的事件委托系统。

```
CDP 点击 → 页面级鼠标事件 → ❌ Vue 代理监听器拦截不到 → handler 不触发
真实点击 → 浏览器 MouseEvent → ✅ Vue 代理监听器拦截 → handler 触发
```

### 解决（首创方案）

直接调用 Vue 内部存储的 handler：

```javascript
const btn = document.querySelector('xhs-publish-btn');
const sym = Object.getOwnPropertySymbols(btn).find(s => s.toString().includes('vei'));
btn[sym].onPublish.value(new Event('click'));
```

备选回退方案（composed:true 突破 Shadow DOM 边界）：
```javascript
btn.dispatchEvent(new MouseEvent('click', {
    bubbles: true, composed: true, cancelable: true
}));
```

## 🕳️ 坑 3：编辑页没有发布按钮

### 现象
在长文编辑页找「发布」按钮，怎么都找不到。

### 原因
小红书长文发布流程不是「编辑 → 发布」，而是：

```
草稿箱 → 编辑 → 一键排版 → 下一步 → 发布页 → 点击发布
```

编辑页底部的 sticky footer 只有「暂存离开」和「一键排版」，**必须经过排版步骤才能到发布页**。

### 解决
严格按照流程：点「一键排版」→ 等加载 → 点「下一步」→ 到发布页 → 发布按钮出现。

## 🕳️ 坑 4：按钮在页面底部看不到

### 现象
用 `document.querySelector` 定位元素时，返回的 `getBoundingClientRect()` 显示 y 坐标很大（如 1580px），但视口只有 1080px。

### 原因
默认视口 1920×1080，但底部 sticky footer 和很多按钮的 y 坐标超出视口。元素虽然在 DOM 中，但不在可视区域内。

### 解决
1. **扩大视口**：
   ```python
   cmd("Emulation.setDeviceMetricsOverride", {
       "width": 1920, "height": 2000,  # 加高
       "deviceScaleFactor": 1, "mobile": False
   })
   ```
2. **主动滚动**：
   ```javascript
   window.scrollTo(0, document.body.scrollHeight);
   ```
3. **scrollIntoView**：
   ```javascript
   el.scrollIntoView({block: 'center'});
   ```

## 🕳️ 坑 5：网慢导致操作超时

### 现象
WebSocket 超时、页面没加载完就开始操作、等待时间不够导致下一步失败。

### 原因
服务器网络不稳定，小红书页面加载慢（尤其是创作者平台）。

### 解决
1. 每个操作后至少 `wait(3-5)` 秒
2. 导航到新页面等 8 秒
3. WebSocket 超时只需重连（Chrome 还活着），不要重启
4. 操作前验证 DOM 存在：
   ```javascript
   document.querySelector('target-element') !== null
   ```

## 🕳️ 坑 6：中文输入无效

### 现象
用 CDP `Input.dispatchKeyEvent` 输入中文，文本框里什么也没有。

### 原因
CDP 的 KeyEvent 只支持 ASCII 字符，中文等非 ASCII 输入需要 IME（输入法）支持，headless Chrome 不支持。

### 解决
不要用 CDP KeyEvent，直接通过 JavaScript 设值：
```javascript
// textarea
ta.value = '中文内容';
ta.dispatchEvent(new Event('input', {bubbles: true}));

// ProseMirror 富文本（小红书用这个）
editor.innerHTML = '<h2>标题</h2><p>正文</p>';
editor.dispatchEvent(new Event('input', {bubbles: true}));
```

## 🕳️ 坑 7：两个 Chrome 抢端口

### 现象
连接 `localhost:9222` 时，操作的是旧的 Chrome 实例，新开的页面看不到。

### 原因
启动新 Chrome 前没有杀掉旧的，两个 Chrome 进程抢同一个 remote-debugging-port。

### 解决
```bash
pkill -f chrome; sleep 2  # 先杀干净
```
注意：`pkill` 和 `google-chrome` 不能放在同一个脚本里（pkill 会误杀脚本自己的 bash 进程）。

## 🕳️ 坑 8：元素匹配到 body

### 现象
用 `textContent.includes('草稿箱')` 匹配元素，返回了 `<body>` 或 `<html>`。

### 原因
`textContent` 会包含子树中所有文本。如果页面任意位置出现了「草稿箱」三个字，body 的 textContent 也会包含它们。

### 解决
过滤条件必须严格：
```javascript
// ❌ 会匹配到 body
el.textContent.includes('草稿箱')

// ✅ 只匹配叶子节点
el.textContent.trim() === '草稿箱' && el.children.length === 0 && el.offsetWidth < 200
```

## 🕳️ 坑 9：Chrome 窗口被放到虚拟桌面

### 现象
VNC 连上后能看到底部 xfce4-panel 面板，但 Chrome 浏览器窗口完全看不到。用 `xdotool` 查却发现 Chrome 窗口存在且有正确的标题，只是不可见。

### 原因
openbox 窗口管理器支持多个虚拟桌面（workspace）。Chrome 启动时可能被自动分配到虚拟桌面 3，而用户看到的是桌面 0。

### 调试
```bash
# 查看窗口在哪个桌面
DISPLAY=:2 xprop -id <窗口ID> _NET_WM_DESKTOP
# → _NET_WM_DESKTOP(CARDINAL) = 3  ← 在桌面3！
```

### 解决
```bash
# 搬回桌面 0
DISPLAY=:2 xdotool set_desktop_for_window <窗口ID> 0
```

## 🕳️ 坑 10：ProseMirror 富文本编辑器 innerHTML 不生效

### 现象
用 `editor.innerHTML = html` 设置内容后，小红书 ProseMirror 编辑器显示仍为空（1 char）。

### 原因
ProseMirror 有自己的内部状态管理，直接设置 `innerHTML` 不会同步到 ProseMirror 的文档模型中。编辑器显示 `is-empty is-editor-empty`。

### 解决
用 `document.execCommand('insertHTML')` 代替：
```javascript
const ed = document.querySelector('.ProseMirror');
ed.focus();
document.execCommand('selectAll', false, null);
document.execCommand('insertHTML', false, '<h2>标题</h2><p>正文</p>');
```

## 🕳️ 坑 11：VNC 验证——AI 看和用户看可能不一样

### 现象
CDP 截图发给你确认，你说看到了，但实际 VNC 里是黑的。我以为是同一个画面，其实不是。

### 原因
CDP `captureScreenshot` 截的是 Chrome 内部的渲染缓冲，不是 VNC 的 X11 画面。Chrome 内部渲染 ≠ VNC 显示。VNC 显示取决于 X11 + GPU/ANGLE + compositor 整条链路。

### 解决
验证 VNC 必须用系统截图工具，不用 CDP：
```bash
DISPLAY=:2 import -window root /tmp/vnc_check.png
```
然后把这张图发给用户确认。

## 🕳️ 坑 12：Chrome VNC 调试方法论

### 黑屏排查 Checklist

```
1. VNC 服务在跑吗？      → ps aux | grep Xtigervnc
2. Chrome 在跑吗？        → pgrep -af chrome | grep -v crashpad
3. Chrome 有窗口吗？      → DISPLAY=:2 xdotool search --name Chrome
4. 窗口在哪个桌面？       → DISPLAY=:2 xprop -id <ID> _NET_WM_DESKTOP
5. Wayland 在捣乱吗？     → 检查 --ozone-platform 参数
6. GPU 进程在崩溃吗？     → tail /tmp/chrome_vnc.log | grep ERROR
7. 系统截图验证           → DISPLAY=:2 import -window root /tmp/test.png
```

### 核心教训

> **不要用 CDP 截图验证 VNC 画面！** CDP 截图是 Chrome 内部渲染，VNC 是 X11 输出，两者不是同一回事。
